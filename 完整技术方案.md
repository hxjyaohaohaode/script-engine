## 三、创意引擎

### 3.1 伏笔设计器——三层伏笔体系

**伏笔Agent**负责，DS-V4-Pro Think Max模式。

```
全剧级伏笔（5-8条）—— 横跨整部剧本
─────────────────────────────────────────────
  埋设在前10%，回收在最后20%，中间3-5次强化
  揭露时产生"恍然大悟"级别的冲击

  设计方法（从终局反推）:
    ① 核心真相有几层含义？（表面/深层/真相）
    ② 每层需要什么信息支撑？
    ③ 信息在什么位置投放？
    ④ 怎么投放才自然（第一遍不注意）？
    ⑤ 怎么投放才不被遗漏（重玩能发现）？
    ⑥ 揭露后重看前文会发现什么新含义？

  产出格式:
    id: F-001
    type: 全剧级
    name: "师父的真实身份"
    surface_layer: "师父总是忘事（玩家以为是性格特点）"
    deep_layer:    "师父从不亲自使用魔法（玩家以为是谦让）"
    truth_layer:   "师父是最强法师，教主角魔法是为了转移记忆"
    plant_scene:   CH01-S03
    reinforce_at:  [CH05-S07, CH12-S02, CH18-S04]
    reveal_scene:  CH25-S05
    wow_factor:    "回忆起师父每次'忘事'的场景，全部有了新含义"

章节级伏笔（20-30条）—— 跨3-5章
─────────────────────────────────────────────
  在A章埋下"看起来不重要但其实是关键"的细节
  在B章通过另一角色视角"强化"
  在C章揭露，产生"串联感"

场景级伏笔（60-100条）—— 章节内
─────────────────────────────────────────────
  对话中嵌入"双关语"——字面意思是一个含义，知道真相后是另一个
  环境描写中藏"线索"——第一遍不会注意，重玩时会发现
  角色行为中埋"矛盾"——表面合理，深想有问题
```

### 3.2 哇塞工厂——五种创意类型

**创意Agent**负责，DS-V4-Pro Think Max模式。

```
类型1: 因果反转
  玩家以为A导致了B，真相是B导致了A
  示例: 玩家以为"城主暴政导致叛军起义"，真相是"叛军暗中操纵导致城主不得不高压"

类型2: 视角欺骗
  玩家以为他看到的就是全部，真相是多个视角的拼合
  设计: 对每个关键事件设计角色A/B/C的多视角版本

类型3: 身份揭露
  角色的所有行为在新身份下有了全新解释
  设计: 表面人设必须在所有已知场景中自洽，真实人设必须比表面人设更合理

类型4: 情感颠覆
  之前所有的情感互动都有了新的含义
  示例: 以为是"师徒关系"，真相是"父子关系"

类型5: 集体误解
  所有角色共同的错误认知被颠覆
  设计: 错误认知如何形成 → 真相如何逐步揭露 → 对所有关系的冲击
```

### 3.3 哇塞工厂工作流程

```
创意Agent (DS-V4-Pro Think Max):

Step 1: 分析世界观核心矛盾的深层含义
  例: "魔法的代价是记忆" →
    记忆可以转移？→ 有人在用别人的记忆？
    记忆可以伪造？→ 有人在篡改历史？
    记忆去了哪里？→ "去处"本身是秘密？

Step 2: 对每个深层含义尝试五种哇塞类型的组合
  "记忆可以转移" + 身份揭露 = "师父的真实身份"
  "记忆可以伪造" + 集体误解 = "所有人都活在虚假记忆中"
  "记忆的去处" + 因果反转 = "学魔法其实在被吸收记忆"

Step 3: 评估每个方案
  可预测性(0-10): 目标3-7（隐约有感但不确定）
  情感冲击(0-10): 目标≥8
  逻辑自洽(0-10): 目标≥8（反转后必须比原来更合理）
  回望价值(0-10): 目标≥7（揭露后重看前文有新发现）

Step 4: 每条伏笔保留2-3个最优方案供人类选择
```

### 3.4 节奏引擎

编排Agent + 创意Agent协作执行。

```
情感曲线——波浪式推进:

  高 ─        ╱╲          ╱╲              ╱╲    ╱╲
     │       ╱  ╲   ╱╲  ╱  ╲      ╱╲    ╱  ╲  ╱  ╲   ╱╲
     │      ╱    ╲ ╱  ╲╱    ╲    ╱  ╲  ╱    ╲╱    ╲ ╱  ╲
  中 ─     ╱      ╲          ╲  ╱    ╲╱              ╲    ╲
     │    ╱                     ╲                       ╲
  低 ────╱
     └────┬────┬────┬────┬────┬────┬────┬────┬────┬────┬──→
         CH1  CH2  CH3  CH4  CH5  CH6  CH7  CH8  CH9  CH10

     ▲ = 哇塞时刻

节奏规则:
  ① 每章至少1个情感峰值（≥7/10）
  ② 连续2个场景≥8 → 必须插入缓冲（≤4）
  ③ 连续3个场景≤4 → 必须提前引爆冲突（≥7）
  ④ 每2-3章至少1个哇塞时刻
  ⑤ 全剧至少3个超级哇塞时刻（≥9）
  ⑥ 开场10%以铺垫为主（≤5）
  ⑦ 最后20%密集回收伏笔，情感持续高位
```

### 3.5 关系织网

创意Agent + 状态Agent协作。

```
第一层: 结构性关系（Phase 2锁定，人类定义）
  核心冲突对 / 表面盟友终将分裂 / 表面敌对终将和解 / 关系枢纽角色

第二层: 动态关系值（Phase 5实时更新，状态Agent维护）
  每对角色: 信任度(0-100) + 好感度(0-100) + 信息差
  更新规则:
    正面互动（帮助/坦诚/牺牲）→ 信任度+好感度上升
    负面互动（欺骗/背叛/伤害）→ 信任度下降
    关系值变化必须有场景支撑

第三层: 关系引爆点（创意Agent设计）
  信任度<20 → 怀疑
  信任度>80 → 愿意为对方冒险
  信任度从80骤降到10 → "背叛"事件（哇塞时刻来源）
```

### 4.3 程序化检测器（代码，成本为零）

```python
# 检测器1: 元素封闭性 — 防瞎编
def check_element_closure(scene_text, registry):
    entities = extract_named_entities(scene_text)
    unregistered = entities - registry
    return Fail(f"未注册元素: {unregistered}") if unregistered else Pass()

# 检测器2: 状态一致性 — 防遗忘
def check_state_consistency(scene, layer1):
    for role in scene.roles:
        state = layer1.get(role.id)
        if role.location != state.location and not scene.has_travel:
            return Fail(f"{role.name}位置不连续")
        if role.emotion != state.emotion and not scene.has_trigger:
            return Fail(f"{role.name}情感突变无触发")
        for info in role.knows:
            if info not in state.known_info:
                return Fail(f"{role.name}不应知道{info}")
    return Pass()

# 检测器3: 伏笔状态机 — 防状态跳跃
LEGAL = {"design":["plant"], "plant":["reinforce","reveal"],
         "reinforce":["reinforce","reveal"], "reveal":["verify"]}
def check_foreshadow_transition(ops, table):
    for op in ops:
        if op.new_status not in LEGAL.get(table[op.fs_id].current_status, []):
            return Fail(f"伏笔{op.fs_id}状态转换非法")
    return Pass()

# 检测器4: 时空连续性 — 防瞬移
def check_spatiotemporal(scenes, new_scene):
    prev = scenes[-1]
    if new_scene.time_start < prev.time_end:
        return Fail("时间倒流")
    if new_scene.location != prev.location:
        if new_scene.time_start - prev.time_end < travel_time(prev.location, new_scene.location):
            return Fail("瞬移")
    return Pass()

# 检测器5: 伏笔回收可达性 — 防断裂
def check_foreshadow_reachability(table, graph):
    broken = [fs_id for fs_id, fs in table.items()
              if fs.current_status in ["plant","reinforce"]
              and not has_path(graph, fs.plant_scene, fs.reveal_scene)]
    return Fail(f"伏笔回收无通路: {broken}") if broken else Pass()

# 检测器6: 关系值连续性 — 防关系突变
def check_relation_continuity(scene, layer1):
    for inter in scene.role_interactions:
        delta = abs(inter.new_relation.trust - layer1.get_relation(inter.a, inter.b).trust)
        if delta > 30 and not scene.has_major_event:
            return Fail(f"{inter.a}-{inter.b}关系值突变{delta}")
    return Pass()
```

## 六、七个Agent的Skills

### 6.1 编排Agent（MiMo-V2-Pro）

**Skill 1: 任务拆解器**

```
输入: 用户需求文档
输出: task_list.yaml

算法:
  1. 解析需求: 题材/风格/字数/单人多人/核心创意
  2. 确定工作模式: 轻量/标准/重型
  3. 生成Phase 0-6任务清单，标注依赖关系和优先级
  输出:
    tasks:
      - id: T-001
        phase: 1
        name: "世界观构建"
        assigned_to: "创作Agent"
        depends_on: []
        priority: 1
```

**Skill 2: 优先级调度器**

```
输入: 全局状态（Layer 1 + 任务队列 + Agent状态）
输出: 下一个待执行任务 + 目标Agent

算法:
  1. 从Layer 4获取当前章节待完成场景
  2. 检查依赖链: 哪些场景的前置已完成？
  3. 锚点场景优先
  4. 有伏笔任务的场景优先
  5. 根据情感曲线判断当前应上升还是下降
  6. 输出优先级最高的场景ID + 创作要求
```

**Skill 3: 冲突仲裁器**

```
输入: 审计Agent封驳 + 创作Agent申辩（或连续3次封驳）
输出: 最终裁决

算法:
  1. 读取封驳原因和修改尝试
  2. 判断: 逻辑问题（需要改）vs 创意评判（可能需放宽）
  3. 逻辑问题 → 要求按审计意见修改
  4. 创意评判 → 降低阈值或标记"需人类介入"
  5. 连续3次封驳 → 通知人类，附带完整上下文
```

**Skill 4: 节奏监控器**

```
输入: 当前章节已完成场景的情感标注
输出: 节奏调整指令

算法:
  1. 统计情感强度序列
  2. 连续2个≥8？→ 下一个必须缓冲（≤4）
  3. 连续3个≤4？→ 下一个必须引爆（≥7）
  4. 本章无哇塞时刻？→ 标记需要
  5. 违反规则 → 向创作Agent下达调整指令
```

### 6.2 创作Agent（DS-V4-Pro）

**Skill 1: 场景撰写器**

```
输入:
  context_pack (素材Agent组装，1-3万字)
  scene_requirements:
    scene_id / scene_type / emotion_target
    foreshadow_tasks: [{fs_id, op: "plant"|"reinforce"|"reveal"}]
    is_wow_moment: bool
    wow_spec: 哇塞方案详细要求（如果是哇塞场景）

输出: scene_draft.yaml

算法:
  1. 选择思考模式:
     普通过渡 → Non-think | 情感冲突 → Think High | 哇塞时刻 → Think Max
  2. 构建prompt:
     【世界观+角色设定】(Layer 0)
     【角色当前状态】(Layer 1)
     【前2个场景全文】(保持文风连续)

  4. 解析为结构化scene_draft:
     scene_id / type / location / weather

     foreshadow_ops: [{fs_id, op, content}]
     causal_chain: {preconditions, catalyst, direct_result, indirect_result, far_result}

```

**Skill 2: 对白生成器**

```
输入: 角色设定 + 当前情感 + 对话对象 + 关系值 + 场景目标
输出: 符合人设的对白

算法:
  1. 从Layer 0读取语言风格/口头禅
  2. 从Layer 1读取情感状态和关系值
  3. 确定对白基调:
     关系高+情感正面 → 亲近信任 | 关系高+情感负面 → 失望受伤
     关系低+情感正面 → 礼貌试探 | 关系低+情感负面 → 敌意防备
  4. 生成: 符合风格 + 有潜台词 + 不同角色有区分度
```

**Skill 3: 分支设计器**

```
输入: 当前场景 + 角色状态 + 伏笔要求
输出: 选择节点

算法:
  1. 分析核心冲突点
  2. 设计2-3个选项:
     → 不是对/错，是不同代价
     → 影响延迟显现
     → 不同选择揭示不同真相
  3. 推演后果: 直接结果 → 间接结果 → 远期结果
  4. 设计隐藏选项（需特定前置条件）
```

### 6.3 审计Agent（DS-Reasoner）

**绝不写任何内容，只做审核。**

**Skill 1: 逻辑审计器**

```
输入: scene_draft + Layer 0-2
输出: audit_report.yaml

阶段A: 程序化检测器（代码，不调用LLM）
  [1] 元素封闭性: 专有名词 vs element_registry
  [2] 状态一致性: scene_draft角色状态 vs Layer 1
  [3] 伏笔状态机: foreshadow_ops状态转换合法性
  [4] 时空连续性: 与前序场景的位置/时间比对
  [5] 伏笔回收可达性: BFS检查plant→reveal通路
  [6] 关系值连续性: 变化幅度>30需重大事件支撑
  任一失败 → 直接返回Fail

阶段B: LLM审计（DS-Reasoner CoT推理）
  [7] 因果链推演:
      "前置条件{preconditions}，从Layer 1验证是否满足"
      "动机{motivation}，从Layer 0人设验证是否合理"
      "催化剂{catalyst}，从Layer 2检索是否有铺垫"
  [8] 人设校验:
      "行为{behavior}，比对行为边界: 必然/绝不/需铺垫"
  [9] 对白风格:
      "对白{dialogue}，比对角色语言风格{style}"

阶段C: 创意评分（仅含伏笔/反转的场景）
  [10] 可预测性: "聪明玩家从已有信息能猜到什么？" → 0-10
  [11] 情感冲击: "场景文本的情感冲击力" → 0-10
  [12] 节奏审计: emotion_level vs 曲线目标 + 节奏规则检查

输出:
  audit_report:
    scene_id: CH03-S07-A
    checks:
      - {name: "元素封闭性", result: "pass"}
      - {name: "因果链推演", result: "fail", detail: "前置条件不满足"}
      - {name: "创意评分", result: "pass", scores: {predictability: 5, emotion: 8}}
    overall: "fail"
    issues: ["因果链前置条件不满足"]
    suggestion: "需要在CH03-S06中补充发现暗格的场景"
```

**Skill 2: 全剧审计器**

```
输入: Layer 0-5全部数据
输出: final_audit_report.yaml

[1] 伏笔回收率: 统计foreshadow_status，目标100%
[2] 时间线自洽: 遍历所有场景时间标记，检测矛盾
[3] 结局可达性: BFS从开局到每个结局，检查路径
[4] 角色弧完整性: 提取角色所有出场场景情感，检查弧线
[5] 情感曲线分析: 绘制全剧曲线，检查连续低谷/高峰
[6] 极端压力测试（重型模式）:

```

### 6.4 状态Agent（MiMo-V2.5-Pro）

**Skill 1: 状态更新器**

```
输入: 定稿的scene_draft
输出: Layer 1增量更新

算法:
  1. 解析状态变化:
     → characters表: emotion/emotion_level/location
     → foreshadow_status表: current_status/reinforce_count
     → info_records表: 新增信息
     → character_relations表: 关系值变化
  2. 自洽检查:
     → 同一角色不能同时在两个地点
     → 关系值不能超出0-100
     → 伏笔状态只能单向前进
  3. 写入PostgreSQL
  4. 广播agent.state.update
```

**Skill 2: 伏笔追踪器**

```
输入: 伏笔总图(Layer 0) + 当前状态(Layer 1) + 已完成场景(Layer 5)
输出: 伏笔健康报告

算法:
  遍历所有伏笔:
    >5章未强化 → "可能被遗忘"
    >10章未回收且未安排回收 → "可能断裂"
    已到回收位置未回收 → "回收遗漏"
  检查伏笔关系层关联:
    伏笔A的reveal依赖伏笔B但B未reveal？
    伏笔间逻辑矛盾？
```

**Skill 3: 关系值管理器**

```
输入: 场景中的角色互动
输出: 更新后的关系值

算法:
  互动类型 → 信任度/好感度变化:
    正面(帮助/坦诚/牺牲): trust += 5-15, favor += 5-10
    中性(对话/合作): trust += 0-3
    负面(欺骗/背叛/伤害): trust -= 10-30
    复杂(表面敌对实际保护): trust -= 5, favor += 10
  修正:
    当前关系高时正面效果+20%（信任基础）
    当前关系低时负面效果+20%（偏见加深）
  引爆点检查:
    trust < 20 → 标记"怀疑"
    trust 从>70骤降到<30 → 标记"背叛事件"，通知编排Agent
```

## 八、完整生产流程

### 8.1 六阶段总览

```
Phase 0  需求对齐              人类+编排Agent         0.5-2天
Phase 1  世界观构建            人类创意+创作Agent     1-5天
Phase 2  角色设计              人类灵魂+创作Agent     1-5天
Phase 3  伏笔网络+哇塞设计     人类创意+伏笔/创意Agent 3-10天
Phase 4  章节大纲+节奏设计     人类框架+创作Agent     2-7天
Phase 5  场景创作              全Agent协作+人类审核   5-80天
Phase 6  全剧终审              审计Agent+人类终审     1-7天
```

### 8.2 Phase 0-2: 需求→世界观→角色

```
Phase 0: 编排Agent拆解需求 → 与用户逐项确认 → 输出需求确认文档

Phase 1:
  人类提出核心矛盾 → 创作Agent推演3个方案(Think Max) → 人类选择
  → 创作Agent细化(Think High) → 审计Agent验证逻辑一致性 → 人类终审 → Layer 0锁定

Phase 2:
  人类定义角色灵魂(动机/恐惧/表面vs真实) → 创作Agent扩展(Think High)
  → 审计Agent验证 → 人类终审 → Layer 0锁定
```

### 8.3 Phase 3: 伏笔网络 + 哇塞设计（核心阶段）

```
Step 1  人类定义核心真相和核心冲突

Step 2  伏笔Agent设计三层伏笔体系（Think Max）
  从核心真相反推 → 全剧级5-8条 + 章节级20-30条 + 场景级60-100条 + 关联图

Step 3  创意Agent设计哇塞方案（Think Max）
  每条全剧级伏笔 → 2-3个方案（因果反转/视角欺骗/身份揭露/情感颠覆/集体误解）
  每个方案附带评分 + 示例场景

Step 4  人类选择最满意的方案

Step 5  审计Agent验证（BFS通路检查 + 逻辑自洽 + 可预测性评估）

Step 6  人类终审 → Layer 0锁定
```

### 8.4 Phase 4: 章节大纲 + 节奏设计

```
Step 1  人类设计全剧骨架（章节数 + 每章核心事件 + 关键转折点）

Step 2  创作Agent展开每章大纲（Think High）
  每章: 核心事件/出场角色/场景数/伏笔任务/分支结构

Step 3  创意Agent设计情感曲线+哇塞节奏（Think High）
  全剧情感曲线 + 哇塞时刻精确位置 + 缓冲/铺垫分布 + 7条节奏规则验证

Step 4  状态Agent锁定Layer 3

Step 5  人类终审
```

### 8.5 Phase 5: 场景创作（单场景循环）

```
Step 1  编排Agent选场景（锚点 > 伏笔任务 > 依赖链最短）

Step 2  素材Agent组装上下文包（1-3万字）

Step 3  创作Agent写场景（Non-think / Think High / Think Max）
  prompt: 情感目标 + 伏笔任务 + 行为边界 + 当前状态 + 前2场景全文 + 哇塞要求

Step 4  审计Agent审核
  阶段A: 程序化检测器6项（代码）
  阶段B: LLM审计6项（DS-Reasoner）

Step 5  判定: 通过→Step 6 / 失败→打回创作Agent / 3次封驳→人类介入

Step 6  人类关键节点审核（按模式裁剪: 哇塞场景/伏笔回收/合流点/结局）

Step 7  状态Agent更新Layer 1 + 素材Agent写入Layer 2(Git) + 更新Layer 5

Step 8  编排Agent推进 → 下一个场景
```

### 8.6 Phase 6: 全剧终审

```
轻量: 伏笔回收率 + 时间线自洽 + 人类终审
标准: 以上 + 结局可达性 + 角色弧 + 情感曲线 + 人类终审
重型: 以上 + 极端压力测试（3种玩家模拟）+ 人类终审

终审后: 审计Agent输出报告 → 创作Agent修复 → 重新审计 → 循环直到通过
```
